`Checkpoint 3.1`: Query Execution Order.

When you write a SQL query, the order of the clauses might seem straightforward (SELECT, then FROM, then WHERE, etc.). However, the database doesn't actually execute them in that written order. Instead, it follows a specific logical processing order.

Understanding this order is crucial for:

- Writing correct queries: Knowing which columns or aliases are available at each step.
- Troubleshooting: Figuring out why your query isn't returning what you expect.
- Optimizing performance: Imagining how the database builds up your result set.

Think of it like following a recipe or an assembly line: you can't frost a cake before it's baked, and you can't bake it before you've mixed the ingredients. SQL queries follow a similar, non-intuitive logical sequence.

The Logical Query Processing Order (Standard SQL):

Here's the typical logical order in which a database processes SQL clauses:

1.  `FROM` (and JOINs, if present):
    - First, the database identifies all the tables specified in the FROM clause. If there are JOINs, it then combines these tables into a single (potentially very large) virtual table. This creates the initial set of rows for the query.
2.  `WHERE`:
    - Next, the database filters this initial set of rows based on the conditions specified in the WHERE clause. Only rows that satisfy these conditions are kept.
3.  `GROUP BY`:
    - The remaining rows are then grouped together based on common values in the columns specified in the GROUP BY clause. This creates
      summary groups (e.g., all sales for a particular product category).
4.  `HAVING`:
    - Similar to WHERE, but HAVING filters groups. Only groups that satisfy the HAVING clause conditions are kept. This typically involves aggregate functions (like COUNT, SUM, AVG).
5.  `SELECT`:
    - Now, the database selects the specific columns or expressions that you want to see in the final output. This is the first point where column aliases (e.g., COUNT(column) AS my_count) are actually established and can be used.
6.  `DISTINCT`:
    - If DISTINCT is specified, duplicate rows are removed from the result set generated by the SELECT clause.
7.  `ORDER BY`:
    - Finally, the remaining rows are sorted according to the criteria specified in the ORDER BY clause.
8.  `LIMIT` / `OFFSET`:
    - The last step is to restrict the number of rows returned, taking a subset of the sorted results. LIMIT specifies the maximum number of rows, and OFFSET specifies how many rows to skip from the beginning.

Key Implications & Common Pitfalls:

- Column Aliases: A column alias defined in the SELECT clause (COUNT(product_id) AS total_products) cannot be used in the WHERE clause
  because WHERE executes before SELECT. However, it can be used in the ORDER BY clause because ORDER BY executes after SELECT.
- Aggregate Functions: Aggregate functions (SUM(), COUNT(), AVG()) are typically used in SELECT (with GROUP BY) or HAVING. They are
  calculated after WHERE filters individual rows.

Example Query (and its logical flow):

1 SELECT category, COUNT(product_id) AS total_products -- Step 5: Select & Alias
2 FROM products -- Step 1: Start with 'products' table
3 WHERE price > 50 -- Step 2: Filter rows where price > 50
4 GROUP BY category -- Step 3: Group remaining rows by 'category'
5 HAVING COUNT(product_id) > 2 -- Step 4: Filter groups where total products > 2
6 ORDER BY total_products DESC -- Step 6: Sort results by alias 'total_products'
7 LIMIT 5;

Checkpoint 3.2: The `SELECT` Statement

1.  Analogy: The Photographer


    Think of yourself as directing a photoshoot. You've already picked the location (FROM) and maybe asked some people to step out of the shot (WHERE).

    Now, you tell the photographer what to capture in the final photo. This is the SELECT statement.
    * SELECT * is like saying, "Just give me the big, wide shot of everything you see."
    * SELECT name, email is like saying, "Zoom in and just get a portrait of each person's face and their name tag."
    * SELECT name AS person is like saying, "Take the portrait, but label it 'person' in the final album."

2. Concept: Choosing Your Columns


    The SELECT clause specifies the columns that will appear in your final result set. It's one of the last parts of the query to be logically
    processed, but it's the first word you type.

    Here are the most common ways to use it:

    * `SELECT *`: The asterisk (*) is a wildcard that means "all columns". It's great for quick exploration, but in real applications, it's
      better to be specific.
    * `SELECT column1, column2`: This is the most common use. You explicitly list the columns you want, separated by commas.
    * `SELECT column AS alias`: The AS keyword lets you rename a column in your output. This is called an alias. It makes results readable
        and is essential for calculated columns.
    * `SELECT DISTINCT column`: The DISTINCT keyword removes duplicate rows from your final result. If you SELECT DISTINCT category, and
        have multiple 'Fruit' items, you will only see 'Fruit' once in your results.

3. Example


    Let's imagine we have a simple employees table:
    ┌────┬────────────┬───────────┬───────────────┐
    │ id │ first_name │ last_name │ department    │
    ├────┼────────────┼───────────┼───────────────┤
    │ 1  │ 'Ana'      │ 'Singh'   │ 'Engineering' │
    │ 2  │ 'Ben'      │ 'Chen'    │ 'Sales'       │
    │ 3  │ 'Chris'    │ 'Davis'   │ 'Sales'       │
    └────┴────────────┴───────────┴───────────────┘

    Query 1: Select all columns
    1 SELECT * FROM employees;
    Result: The entire table as shown above.

    Query 2: Select specific columns
    1 SELECT first_name, department FROM employees;
    Result:
    ┌────────────┬───────────────┐
    │ first_name │ department    │
    ├────────────┼───────────────┤
    │ 'Ana'      │ 'Engineering' │
    │ 'Ben'      │ 'Sales'       │
    │ 'Chris'    │ 'Sales'       │
    └────────────┴───────────────┘

    Query 3: Use an alias
    1 SELECT first_name AS name, department AS dept FROM employees;
    Result: The column headers are now name and dept.
    ┌─────────┬───────────────┐
    │ name    │ dept          │
    ├─────────┼───────────────┤
    │ 'Ana'   │ 'Engineering' │
    │ 'Ben'   │ 'Sales'       │
    │ 'Chris' │ 'Sales'       │
    └─────────┴───────────────┘

    Query 4: Select distinct departments
    1 SELECT DISTINCT department FROM employees;
    Result: Notice 'Sales' only appears once.
    | department   |
    |--------------|
    | 'Engineering'|
    | 'Sales'      |

4. Summary
   SELECT is how you define the final shape of your data. You can grab everything with \*, pick specific columns, rename them with AS for
   clarity, and use DISTINCT to filter out duplicates.

Checkpoint 3.3: The `WHERE` Clause

1. Analogy: The Gatekeeper
   Imagine you're hosting a party, and you've got a guest list (your FROM table). Before anyone even gets inside the party to be seen
   (SELECT) or grouped into conversations, there's a gatekeeper at the door.
   The WHERE clause is that gatekeeper. It checks each guest against a set of rules, and only the ones who meet the criteria are allowed in. "Only people who RSVP'd" (a condition on a column), or "people wearing formal attire AND bringing a dish" (multiple conditions).

2. Concept: Filtering Rows
   The WHERE clause is used to filter rows based on a specified condition. It's one of the earliest clauses to be processed logically in a query, right after FROM. This means that only the rows that satisfy the WHERE condition are passed on to the next stages of the query (like GROUP BY or SELECT).


    Here are the common oper ators you'll use:
    * Comparison Operators:
        * = (Equal to)
        * != or <> (Not equal to)
        * > (Greater than)
        * < (Less than)
        * >= (Greater than or equal to)
        * <= (Less than or equal to)
    * Logical Operators: Combine conditions
        * AND: Both conditions must be true.
        * OR: At least one condition must be true.
        * NOT: Negates a condition.
    * Special Operators:
        * BETWEEN value1 AND value2: Checks if a value is within a range (inclusive).
        * LIKE 'pattern%': Used for pattern matching (e.g., 'A%' for anything starting with 'A').
        * IN (value1, value2, ...): Checks if a value matches any value in a list.
        * IS NULL: Checks for NULL values.
        * IS NOT NULL: Checks for non-NULL values.

3. Example
   Let's use our students table again:
   ┌────────────┬────────────┬───────────┬────────────────────┬─────────────┐
   │ student_id │ first_name │ last_name │ major │ year │
   ├────────────┼────────────┼───────────┼────────────────────┼─────────────┤
   │ 1 │ 'Alice' │ 'Smith' │ 'Computer Science' │ 'Freshman' │
   │ 2 │ 'Bob' │ 'Johnson' │ 'Mathematics' │ 'Sophomore' │
   │ 3 │ 'Charlie' │ 'Brown' │ 'Computer Science' │ 'Freshman' │
   │ 4 │ 'Diana' │ 'Prince' │ 'Physics' │ 'Junior' │
   │ 5 │ 'Eve' │ 'Adams' │ 'Mathematics' │ 'Sophomore' │
   └────────────┴────────────┴───────────┴────────────────────┴─────────────┘


    Query 1: Students in 'Computer Science'
    1 SELECT * FROM students WHERE major = 'Computer Science';
    Result:
    ┌────────────┬────────────┬───────────┬────────────────────┬────────────┐
    │ student_id │ first_name │ last_name │ major              │ year       │
    ├────────────┼────────────┼───────────┼────────────────────┼────────────┤
    │ 1          │ 'Alice'    │ 'Smith'   │ 'Computer Science' │ 'Freshman' │
    │ 3          │ 'Charlie'  │ 'Brown'   │ 'Computer Science' │ 'Freshman' │
    └────────────┴────────────┴───────────┴────────────────────┴────────────┘

    Query 2: Sophomores or Juniors
    1 SELECT first_name, year FROM students WHERE year = 'Sophomore' OR year = 'Junior';
    Result:
    ┌────────────┬─────────────┐
    │ first_name │ year        │
    ├────────────┼─────────────┤
    │ 'Bob'      │ 'Sophomore' │
    │ 'Diana'    │ 'Junior'    │
    │ 'Eve'      │ 'Sophomore' │
    └────────────┴─────────────┘

    Query 3: Students whose first name starts with 'A'
    1 SELECT * FROM students WHERE first_name LIKE 'A%';
    Result:
    ┌────────────┬────────────┬───────────┬────────────────────┬────────────┐
    │ student_id │ first_name │ last_name │ major              │ year       │
    ├────────────┼────────────┼───────────┼────────────────────┼────────────┤
    │ 1          │ 'Alice'    │ 'Smith'   │ 'Computer Science' │ 'Freshman' │
    └────────────┴────────────┴───────────┴────────────────────┴────────────┘

    Query 4: Students with IDs between 2 and 4 (inclusive)
    1 SELECT first_name, student_id FROM students WHERE student_id BETWEEN 2 AND 4;
    Result:
    ┌────────────┬────────────┐
    │ first_name │ student_id │
    ├────────────┼────────────┤
    │ 'Bob'      │ 2          │
    │ 'Charlie'  │ 3          │
    │ 'Diana'    │ 4          │
    └────────────┴────────────┘

4. Summary
   The WHERE clause is your primary tool for filtering individual rows from your table. It comes early in the query process and uses
   various comparison, logical, and special operators to define which rows make it into the next stages.

Checkpoint 3.4: The `INSERT` Statement

1. Analogy: Adding a New Entry
   If SELECT is like reading existing information from a file, INSERT is like writing a brand new entry into that file. You're not
   changing what's already there; you're just adding a fresh piece of information, creating a new row in your table.

2. Concept: Adding New Rows to a Table
   The INSERT statement is used to add one or more new rows (records) into a table. This is how you populate your database with data.
   There are a couple of common ways to use INSERT:


    * Syntax 1: Specifying Columns (Best Practice)
    1 INSERT INTO table_name (column1, column2, column3, ...)
    2 VALUES (value1, value2, value3, ...);
      * You explicitly list the columns you are providing values for in parentheses.
      * The VALUES clause then lists the corresponding values for those columns, in the same order.
      * Why is this best practice? It makes your queries more robust. If someone adds a new column to the table later, or changes the
        order of columns, your INSERT statement will still work as long as you're providing values for the columns you listed.
      * Important: Text values (strings) must be enclosed in single quotes (' '). Numbers and boolean values generally do not need quotes.

    * Syntax 2: Omitting Columns (Use with Caution)
    1 INSERT INTO table_name
    2 VALUES (value1, value2, value3, ...);
      * With this syntax, you must provide a value for every column in the table, and in the exact order the columns were defined when the
        table was created.
      * This is generally not recommended for production code because it's fragile: any change to the table's schema (adding, removing, or
       reordering columns) will break this query.

    * Inserting Multiple Rows at Once (Efficient!)
    1     INSERT INTO table_name (column1, column2)
    2     VALUES
    3     (value1a, value2a),
    4     (value1b, value2b),
    5     (value1c, value2c);
    * You can add multiple rows in a single INSERT statement by separating each set of VALUES with a comma. This is often more
      efficient than running many individual INSERT statements.
    * Considerations:
        * Primary Keys: If a table has a primary key that is automatically generated (like SERIAL in PostgreSQL), you usually omit that
          column from your INSERT list, or use the DEFAULT keyword for its value.
        * `NOT NULL` Constraints: You must provide a value for any column defined with a NOT NULL constraint.
        * Default Values: If a column has a DEFAULT value defined, you can omit that column from your INSERT statement, and the database
          will automatically use its default.

3. Example
   Let's use a simple products table:
   ┌────────────┬──────┬───────┬──────────┐
   │ product_id │ name │ price │ category │
   ├────────────┼──────┼───────┼──────────┤
   │ (auto) │ │ │ │
   └────────────┴──────┴───────┴──────────┘
   (Assume product_id is an auto-incrementing primary key and category has a default value of 'General'.)


    Query 1: Inserting a single product, specifying columns
    1 INSERT INTO products (name, price)
    2 VALUES ('Laptop', 1200.00);
    After this, a new row would exist, for example: `(1, 'Laptop', 1200.00, 'General')`

    Query 2: Inserting a single product, specifying all columns (including a different category)
    1 INSERT INTO products (name, price, category)
    2 VALUES ('Mouse', 25.00, 'Electronics');
    Example new row: `(2, 'Mouse', 25.00, 'Electronics')`

    Query 3: Inserting multiple products
    1 INSERT INTO products (name, price, category)
    2 VALUES
    3 ('Keyboard', 75.00, 'Electronics'),
    4 ('Monitor', 300.00, 'Electronics');
    Example new rows: `(3, 'Keyboard', 75.00, 'Electronics')`, `(4, 'Monitor', 300.00, 'Electronics')`

4. Summary


    The INSERT statement is how you add new data into your tables. Always specifying the column list is the most reliable way to write
    INSERT queries, and you can insert multiple rows efficiently within a single statement.

Checkpoint 3.5: The `UPDATE` Statement

1. Analogy: Editing an Existing Document
   If INSERT is like adding a brand new page to a document, UPDATE is like going back to an existing page and making corrections or
   changes to the information already written there. You're not adding new content; you're modifying what's already present.

2. Concept: Modifying Existing Rows in a Table
   The UPDATE statement is used to change existing records in a table. You specify which table to update, which columns to change, what
   their new values should be, and most importantly, which specific rows to apply these changes to.


    * Syntax:
    1 UPDATE table_name
    2 SET column1 = new_value1,
    3 column2 = new_value2,
    4         ...
    5 WHERE condition;
      * `UPDATE table_name`: Identifies the table where changes will occur.
      * `SET column1 = new_value1, ...`: Specifies the column(s) you want to modify and the new value(s) they should hold. You can update
        one or many columns in a single SET clause.
      * `WHERE condition`: This is the most critical part! The WHERE clause determines which specific rows will be updated. If you omit
        the WHERE clause, the UPDATE statement will modify every single row in the table_name! Always double-check your WHERE condition to
        ensure you're targeting only the intended rows.
      * New values can be literal values, results of expressions, or even values from other columns.

3. Example
   Let's use our books table. Assume it currently looks like this after our previous INSERT operations:
   ┌─────────┬─────────────────────────┬───────────────────────┬─────────────┬──────────┬────────┐
   │ book_id │ title │ author │ genre │ pub_year │ rating │
   ├─────────┼─────────────────────────┼───────────────────────┼─────────────┼──────────┼────────┤
   │ 1 │ 'The Great Gatsby' │ 'F. Scott Fitzgerald' │ 'Fiction' │ 1925 │ 4.3 │
   │ 2 │ 'To Kill a Mockingbird' │ 'Harper Lee' │ 'Fiction' │ 1960 │ NULL │
   │ 3 │ 'Pride and Prejudice' │ 'Jane Austen' │ 'Romance' │ 1813 │ NULL │
   │ 4 │ 'Animal Farm' │ 'George Orwell' │ 'Dystopian' │ 1945 │ 4.0 │
   └─────────┴─────────────────────────┴───────────────────────┴─────────────┴──────────┴────────┘


    Query 1: Update the rating for 'To Kill a Mockingbird'
    1 UPDATE books
    2 SET rating = 4.5
    3 WHERE title = 'To Kill a Mockingbird';
    After this query, the row for 'To Kill a Mockingbird' would have rating = 4.5.

    Query 2: Update the genre and publication year for 'Animal Farm'
    (Let's pretend we made a mistake and its genre should be 'Allegory' and pub_year was actually 1946)
    1 UPDATE books
    2 SET genre = 'Allegory', pub_year = 1946
    3 WHERE title = 'Animal Farm';
    After this, the row for 'Animal Farm' would have genre = 'Allegory' and pub_year = 1946.

    Query 3: Increase the rating for all books published before 1930 by 0.2
    1 UPDATE books
    2 SET rating = rating + 0.2
    3 WHERE pub_year < 1930;
    This would affect 'The Great Gatsby' (1925) and 'Pride and Prejudice' (1813, if it had a rating). 'The Great Gatsby's rating would change from 4.3 to 4.5.

    🚨 DANGER EXAMPLE 🚨: What happens if you forget `WHERE`?
    1 UPDATE books
    2 SET rating = 5.0;
    If you ran this, every single book in your table would suddenly have a rating of 5.0! Always use WHERE to target your changes.

4. Summary
   The UPDATE statement is used to change existing data in your tables. You specify which columns to SET to new values. The WHERE clause
   is absolutely essential to ensure you only modify the intended rows; omitting it will update every row in the table!

Checkpoint 3.6: The `DELETE` Statement

1. Analogy: Tearing Pages Out
   If INSERT adds new pages and UPDATE lets you edit existing pages, then DELETE is like physically tearing out entire pages (or
   sections of pages) from your book. The data is gone, and unless you have a backup, it's very difficult to get it back.

2. Concept: Removing Existing Rows from a Table
   The DELETE statement is used to remove one or more existing rows from a table. Just like UPDATE, the WHERE clause is absolutely
   crucial here.


    * Syntax:
    1 DELETE FROM table_name
    2 WHERE condition;
      * `DELETE FROM table_name`: Specifies the table from which rows will be removed.
      * `WHERE condition`: This is the most dangerous and critical part! The WHERE clause determines which specific rows will be deleted.
        If you omit the WHERE clause, the DELETE statement will remove every single row from your table_name! This action is often
        irreversible without database backups.

    * Important Distinction (`DELETE` vs. `TRUNCATE` vs. `DROP`):
      * `DELETE`: Removes rows one by one. It's slower for very large tables but can be used with a WHERE clause to target specific rows.
        It often logs each deletion, making it potentially reversible (if done within a transaction).
      * `TRUNCATE TABLE`: Removes all rows from a table much faster than DELETE without a WHERE clause, especially for large tables. It
        typically does not log individual row deletions and cannot be rolled back easily. It resets auto-incrementing counters. It keeps
        the table structure intact.
      * `DROP TABLE`: Removes the entire table from the database, including its structure (schema) and all its data. This is the most
        destructive operation.

    For this checkpoint, we're focusing purely on DELETE FROM ... WHERE ....

3. Example
   Let's use our books table again. Assume it looks like this:
   ┌─────────┬─────────────────────────┬───────────────────────┬─────────────┬──────────┬────────┐
   │ book_id │ title │ author │ genre │ pub_year │ rating │
   ├─────────┼─────────────────────────┼───────────────────────┼─────────────┼──────────┼────────┤
   │ 1 │ 'The Great Gatsby' │ 'F. Scott Fitzgerald' │ 'Fiction' │ 1925 │ 4.3 │
   │ 2 │ 'To Kill a Mockingbird' │ 'Harper Lee' │ 'Fiction' │ 1960 │ 4.5 │
   │ 3 │ 'Pride and Prejudice' │ 'Jane Austen' │ 'Romance' │ 1813 │ 4.8 │
   │ 4 │ 'Animal Farm' │ 'George Orwell' │ 'Allegory' │ 1946 │ 4.0 │
   │ 5 │ '1984' │ 'George Orwell' │ 'Dystopian' │ 1949 │ 4.5 │
   └─────────┴─────────────────────────┴───────────────────────┴─────────────┴──────────┴────────┘


    Query 1: Delete a specific book by title
    1 DELETE FROM books
    2 WHERE title = 'The Great Gatsby';
    After this, the row for 'The Great Gatsby' would be permanently removed.

    Query 2: Delete all books published before a certain year
    1 DELETE FROM books
    2 WHERE pub_year < 1900;
    This would remove 'Pride and Prejudice' and any other books matching the condition.

    🚨 EXTREME DANGER EXAMPLE 🚨: What happens if you forget `WHERE`?
    1 DELETE FROM books;
    If you ran this query, every single book record would be permanently deleted from your books table! This is almost always a mistake unless you truly
    intend to empty the entire table.

4. Summary
   The DELETE statement permanently removes rows from a table. The WHERE clause is absolutely vital to specify which rows to remove.
   Forgetting the WHERE clause will result in the deletion of all rows, which is a highly destructive and often irreversible action.
